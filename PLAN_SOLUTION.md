# План решения домашнего задания (LLMpro_hw_4_Kuzin_Egor.ipynb)

## 1) Декомпозиция задания по блокам

1. **Постановка и метрики бизнеса (разделы 2–4)**
   - Сформулировать 2+ бизнес-метрики (например, скорость маршрутизации обращения, снижение нагрузки на поддержку, стоимость разметки).
   - Свести задачу к ML-классификации: объект — текст сообщения клиента, метка — intent/тема обращения.
   - Зафиксировать ML-метрики для сравнения LLM и крауда: accuracy, macro-F1, coverage, format compliance/error rate, agreement.

2. **Данные и baseline LLM-разметка (раздел 5)**
   - Загрузить датасет, проверить распределение классов, размеры train/test, редкие классы.
   - Подготовить baseline prompt с явным списком классов и форматом ответа JSON `{"label": "..."}`.
   - Реализовать robust-парсер ответа LLM (обработка «грязного» JSON, лишнего текста, неизвестных меток).
   - Запустить разметку на подвыборке/тесте и собрать baseline-метрики.

3. **Улучшение prompt engineering (раздел 6)**
   - Добавить **few-shot** примеры только из train (без утечки теста).
   - Добавить **CoT**-формат `{"reasoning": "...", "label": "..."}` и обновить парсер.
   - Реализовать минимум одно дополнительное улучшение (например, self-consistency: несколько прогонов + голосование).
   - Сравнить с baseline и подтвердить улучшение accuracy (цель — побить baseline 0.6).

4. **Оценка уверенности (раздел 7)**
   - Реализовать функцию расчёта confidence по среднему token logprob ответа.
   - Проверить связь confidence и корректности ответа (AUC по задаче «верно/неверно»).
   - Визуализировать/таблично показать, что при росте confidence растёт вероятность корректной метки.

5. **Human-in-the-loop (раздел 8)**
   - Реализовать симуляцию шумного разметчика (`simulate_noisy_human`).
   - Реализовать гибридный алгоритм (`simulate_hybrid`) с порогом confidence.
   - Просканировать thresholds 0..1 (шаг 0.05), посчитать accuracy/coverage.
   - Выбрать рабочий порог по правилу: accuracy ≥ целевого уровня, coverage максимален.
   - Сравнить 3 режима: human-only, llm-only, hybrid-best + краткие выводы.

## 2) Практический порядок выполнения в ноутбуке

1. **Сначала заполнить текстовые ответы** в разделах 2–4 (быстро закрывает теорию и критерии оценки).
2. **Сделать baseline-инференс** на ограниченном числе примеров (10–20) для отладки prompt+парсера.
3. **После отладки — прогон на полном eval-срезе** и фиксация baseline-таблицы.
4. **Итеративно улучшать prompt** (few-shot → CoT → одно дополнительное улучшение), каждый раз логируя метрики.
5. **Реализовать confidence** и проверить AUC.
6. **Реализовать human-in-the-loop блок** и подобрать порог.
7. **Собрать финальную сводку**: таблица метрик по всем версиям + краткие бизнес-выводы.

## 5) Критерии «хорошего» итогового решения

- **Корректность**: все требуемые функции реализованы и дают ожидаемые значения.
- **Воспроизводимость**: одинаковые результаты при повторном запуске.
- **Интерпретируемость**: понятна причина улучшений (через аналитику ошибок и reasoning).
- **Практичность**: выбранный threshold даёт разумный баланс качества и автоматизации.
